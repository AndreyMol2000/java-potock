2. Анализ производительности приложения (VisualVM)

Для анализа работы приложения использовался инструмент VisualVM, который позволяет отслеживать использование CPU, памяти и состояние потоков JVM.

Мониторинг использования ресурсов

В процессе работы приложения был проведён мониторинг через вкладку Monitor.

В ходе наблюдения были получены следующие результаты:

Использование CPU оставалось на низком уровне (в пределах нескольких процентов), так как основная задача приложения выполняется периодически с помощью планировщика.

Использование памяти находилось примерно в диапазоне 30–40 MB heap.

Количество потоков JVM составляло около 45–50 потоков, включая системные потоки JVM, потоки сервера Tomcat и пользовательские потоки приложения.

Рост использования памяти наблюдается во время обработки XML-данных и создания объектов, после чего память частично освобождается сборщиком мусора.

Анализ потоков (Thread Dump)

Для анализа работы потоков был выполнен Thread Dump.

В результате были обнаружены следующие типы потоков:

1. Поток планировщика

scheduling-1

Этот поток создаётся Spring Scheduler и отвечает за запуск метода, помеченного аннотацией @Scheduled.

Именно этот поток инициирует процесс получения курсов валют и запуска задач обработки.

2. Пул потоков приложения

pool-2-thread-1
pool-2-thread-2
pool-2-thread-3
pool-2-thread-4
pool-2-thread-5

Данные потоки принадлежат ExecutorService, который используется для параллельной обработки валют.

Каждая валюта (USD, EUR, GBP, CNY, JPY) обрабатывается отдельной задачей, что позволяет выполнять парсинг и сохранение данных параллельно.

Большинство потоков находились в состоянии:

WAITING
TIMED_WAITING

Это означает, что потоки ожидают поступления новых задач и не создают дополнительной нагрузки на процессор.

3. Потоки веб-сервера

http-nio-8080-exec-*

Эти потоки принадлежат встроенному серверу Tomcat и используются для обработки HTTP-запросов.

4. Потоки сетевого клиента

reactor-http-nio-*

Эти потоки используются библиотекой WebClient (Netty) для выполнения сетевых запросов к серверу Центрального Банка России.

Анализ памяти (Heap Dump)

Для анализа использования памяти был создан Heap Dump.

Результаты анализа показали, что наибольшее количество памяти занимают следующие типы объектов:

byte[]

java.lang.String

объекты из пакета java.util.concurrent

Это связано с тем, что при обработке XML-документа создаётся большое количество строковых объектов и временных буферов.

Общий размер heap составлял примерно 37 MB, что является нормальным показателем для небольшого Spring Boot приложения.

Работа сборщика мусора

Во время мониторинга наблюдалась характерная пилообразная форма графика heap памяти.

Это означает:

память увеличивается во время обработки данных

после этого выполняется Garbage Collector (G1 GC)

неиспользуемые объекты удаляются из памяти

Паузы сборщика мусора были короткими и не влияли на работу приложения.

Вывод

Проведённый анализ показал, что приложение работает стабильно:

CPU используется минимально

память используется в пределах нормы

потоки корректно распределяются между планировщиком, пулом задач и сервером

утечек памяти не обнаружено

Основная нагрузка приложения связана с парсингом XML и созданием строковых объектов, что является ожидаемым поведением для данного типа задачи.

3. Анализ использования памяти и работы сборщика мусора

Для анализа памяти использовались данные из VisualVM и метрики JVM, доступные через Spring Boot Actuator (/actuator/prometheus).

Использование памяти

Во время работы приложения наблюдалось использование heap-памяти примерно в диапазоне 30–40 MB.

Основной рост памяти происходил в момент выполнения задачи планировщика, когда приложение:

получает XML-файл с курсами валют с сайта Центрального банка;

парсит XML;

создаёт объекты Rate;

сохраняет данные в базу.

В этот момент создаётся большое количество временных объектов (строки, DOM-элементы XML), поэтому использование памяти временно увеличивается.

После завершения обработки часть памяти освобождается сборщиком мусора.

Работа сборщика мусора

В приложении используется G1 Garbage Collector, который является стандартным сборщиком мусора для современных версий Java.

Во время мониторинга наблюдалась характерная динамика использования памяти:

память постепенно увеличивается во время обработки данных;

затем происходит запуск сборщика мусора;

после этого часть памяти освобождается.

Это отражается на графике памяти в виде пилообразной формы.

Паузы сборщика мусора были короткими и не вызывали заметных задержек в работе приложения.

Основные объекты в памяти

Согласно анализу heap dump, наибольшее количество памяти занимают:

byte[]

java.lang.String

объекты из XML DOM

Это связано с тем, что XML-документ содержит большое количество строковых данных, которые временно создаются при парсинге.

Вывод

Анализ памяти показал, что приложение использует ресурсы эффективно:

использование памяти остаётся стабильным;

сборщик мусора корректно освобождает неиспользуемые объекты;

признаков утечек памяти обнаружено не было.

Основная нагрузка на память возникает во время парсинга XML и создания строковых объектов, что является ожидаемым поведением для данного типа задачи.
